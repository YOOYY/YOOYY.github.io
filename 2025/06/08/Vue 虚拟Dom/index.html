Vue1的响应式弊端

在Vue1的版本中，每次响应式数据发生变化，就会对相应的dom进行更新操作，下面是其简易实现：

每一次 data 数据的更新，Vue都会直接修改真实 dom ，这样虽然能精准追踪数据的变化，进行最小颗粒度的更新，但是在响应式数据很多的情况下，会创建很多的 Watcher ，也会产生性能损耗，所以在Vue2中将依赖收集的颗粒度调整到了以组件为单位，即在挂载组件时创建一个 Watcher 对象，组件中任意响应式数据的变化都只收集和通知这一个组件 Watcher ，这样就避免了 Watcher 过多的问题。

组件如何比较变化

解决了 Watcher 过多的问题，却又引入了新的问题，组件怎么进行差异比较？
答案是：虚拟 dom + diff 算法，这种策略在 react 中已经有所应用而且效果不错。通过生成虚拟 dom 构建虚拟 dom 树，通过 diff 比较新老 dom 树的差异进行高效的 dom 更新。

虚拟dom是做什么的

虚拟dom本质就是一个普通的js对象，和真实 dom 功能类似，也是描述 dom ，记录 dom 的结构关系。那为什么不用真实 dom 呢？因为真实dom很费性能，一方面真实 dom 有很多没必要比较的属性，另一方面， js 和 dom 的交互本来就不是高效的操作，应当尽可能减少这种交互。

虚拟dom是怎么生成的

在SFC中 vnode 一般是通过组件的 render 函数生成的，template中写的 dom 结构都会被编译成组件的 render 函数，这个函数会调用 createElement 和 createTextVNode 等创建虚拟 dom 并返回

当然你也可以直接书写 render 函数， render 函数的 h 方法的全称是 createElement , 也就是上面的 createElement ； createElement 中会根据 tag 区分组件和普通dom元素；组件中会继续区分异步组件，函数组件和类组件，而 createTextVNode 会在 normalizeChildren 中被调用

vnodeData 是从哪来的

在SFC中一般是直接编译得到的，如果是自定义 render 函数，则是 h(tag,data,children) 中的 data
