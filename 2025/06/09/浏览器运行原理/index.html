浏览器运行原理

发表于2025-06-09|更新于2025-06-16|浏览器
|浏览量:

进程和线程

进程是cpu资源分配的最小单位，是能拥有资源和独立运行的最小单位。一个进程就是一个程序的运行实例。
线程是cpu调度的最小单位，线程是建立在进程的基础上的一次程序运行单位。线程是不能单独存在的，它是由进程来启动和管理的。
进程和线程之间的关系有以下 4 个特点：

  1. 进程中的任意一线程执行出错，都会导致整个进程的崩溃
  2. 线程之间共享进程中的数据
  3. 当一个进程关闭之后，操作系统会回收进程所占用的内存
  4. 进程之间的内容相互隔离

多进程浏览器

组成

  • Browser进程(浏览器进程)，浏览器的主进程，主要包括以下功能：
    • 子进程管理，创建，销毁和协调其他进程，提供IPC通信
    • 绘制用户界面，主要包括：地址栏，后退/前进按钮，书签目录等，不包括网页窗口；以及将从渲染进程得到的内存中的Bitmap（位图），绘制到用户界面上
    • 数据存储，在硬盘中保存类似cookie,local storage的各种数据
    • 用户交互
  • 浏览器渲染进程（浏览器内核）：
    • 默认每个Tab页面一个进程，互不影响。出于安全考虑，渲染进程都是运行在沙箱模式下
    • 内部是多线程的，渲染引擎和 JavaScript 引擎都是运行在该进程中
    • 任务是页面渲染（将 HTML、CSS 和 JavaScript 转换为用户可以与之交互的网页），脚本执行，事件处理等
    • Internet Explorer 使用 Trident；Firefox 使用 Gecko、Safari 使用 WebKit；Chrome ，Edge 和 Opera使用 Blink，即 WebKit 的一个分支
  • GPU进程：最多一个，合成分层位图，3D绘制等
  • 网络进程：主要负责页面的网络资源加载
  • 第三方插件进程：每种类型的插件对应一个进程，仅当使用该插件时才创建

浏览器多进程的优势

  • 稳定，避免单个页面或插件崩溃影响整个浏览器
  • 流畅，单个页面的复杂计算不会影响其他页面
  • 多进程充分利用多核优势
  • 方便使用沙盒模型隔离插件等进程，提高浏览器安全性

渲染进程组成

  • 主线程 (Main)：负责 HTML/CSS 解析，JS 执行，样式计算（Recalculate Style），布局（Layout），生成绘制列表（Paint）, 主要调用的模块包括渲染引擎和js引擎
    • JS引擎负责解析Javascript脚本，运行代码，主要有V8（Chrome / Edge）、SpiderMonkey（firefox）、JavaScriptCore（safari）
    • 渲染引擎负责渲染浏览器界面，主要有Blink（Chrome / Edge）、Gecko（firefox）、Webkit（safari）
  • 合成线程 (Compositor)： 合成图层（Layer），处理滚动、动画等“非主线程绘制”
  • 光栅线程 (Raster)：将主线程或合成线程生成的绘制指令（Display List），转化为像素图（位图）；支持并发，多线程运行
  • GPU 线程：接收光栅线程生成的位图，调用 OpenGL 或 Vulkan 等 GPU 接口进行合成显示
  • 输入/事件线程（Input Event Handling Thread）：接收和初步处理鼠标、键盘、滚动等用户输入事件；比如使用 passive: true 时，表示开发者不会调用 event.preventDefault()，浏览器就可以让合成线程提前处理滚动，从而提升滑动流畅度。

浏览器缓存机制

缓存资源分类

目前Webkit资源分成两类，一类是主资源，比如HTML页面，或者下载项，一类是派生资源，比如HTML页面中内嵌的图片或者脚本链接，分别对应代码中两个类：MainResourceLoader和SubresourceLoader。

缓存位置

MemoryCache（内存缓存）

缓存策略：主要由浏览器内部策略管理，不受 Cache-Control: no-store 以外的缓存首部的影响；
优先考虑的资源类型：GET 请求、预加载资源、图片、脚本、样式等静态资源
清除策略：标签页关闭、内存紧张、LRU淘汰策略
特点：读取高效，缓存持续性很短

Service Worker

是一种在浏览器后台运行的独立线程，常用于 拦截请求、缓存资源、离线访问 等高级功能，是构建 PWA（渐进式 Web 应用）的核心之一

diskCache（硬盘缓存）

缓存策略：由后端的 HTTP 响应头严格控制
优先考虑的资源类型：GET 请求、预加载资源、图片、脚本、样式等静态资源
特点：读取速度慢，比之 Memory Cache胜在容量和存储时效性上。

Push Cache（推送缓存）

HTTP/2 中的内容，当以上三种缓存都没有命中时，它才会被使用。一旦连接被关闭，Push Cache 就被释放，并且缓存时间也很短暂，在Chrome浏览器中只有5分钟左右，同时它也并非严格执行HTTP头中的缓存指令。

缓存分类

强缓存

  • Cache-control（HTTP 1.1）

    • max-age：资源在本地缓存的时间，单位为秒(s)，若max-age=0，则表示是一个立即过期的响应（直接标记为陈旧状态）。
    • no-cache：客户端可以缓存资源，但每次使用前必须向服务器确认资源是否过期。即每次都要发请求给服务器询问缓存是否有变化，如果没有变化，那么服务器换返回304 Not Modified状态码，可以直接使用缓存的内容。如果有变化，那么必须从服务器请求新的数据。
    • no-store：禁止浏览器以及所有中间缓存存储任何版本的返回响应,每次用户请求时,都会向服务器发送请求,并下载完整的响应。
    • must-revalidate：一旦缓存过期，就必须向服务器重新验证，如果无法验证（如离线），就不能使用旧缓存。
    • public：任何途径的缓存者(本地缓存，代理服务器），可以无条件的缓存该资源，大多数情况下,public不是必须的。
    • private：浏览器可以缓存private响应。这些响应通常只为单个用户缓存,因此不允许任何中间缓存对其进行缓存,例如,用户的浏览器可以缓存包含用户私人信息的HTML网页,但CDN不能缓存。
    • 这些值可以混合使用，例如 Cache-control:public, max-age=2592000。

  • Expires（HTTP 1.0）
    功能：在规定的时间之前，浏览器不会发出请求，而是直接使用本地缓存
    格式：Expires: Thu, 01 Dec 1994 16:00:00 GMT （必须是GMT格式），返回的时间为服务器端时间

  • Pragma：no-cache（HTTP 1.0）：禁止缓存

  • 优先级：Pragma > Cache-Control > Expires

协商缓存

  • If-None-Match/ETag（HTTP 1.1）
    这两个是成对出现的，浏览器的头部是If-None-Match:”856246825”,取值为上一次服务器返回的ETag的值。
    服务端的头部是ETag:”856246825”,表示资源的摘要值，如果服务器发现资源的最新摘要值跟if-None-Match不匹配，就会返回新的资源（200 OK），否则，就不会返回资源的具体数据（304 Not Modified）
    不同类型的Web服务器生成ETag的策略以及生成的格式是不同的，Etag 在资源分布在多台机器上时，对于同一个资源，不同服务器生成的Etag可能不相同，此时就会导致304协议缓存失效，客户端还是直接从server取资源。可以自己修改服务器端etag的生成方式，根据资源内容生成同样的etag。

  • If-Modified-Since/Last-Modified（HTTP 1.0）
    这两个是成对出现的，浏览器的头部是If-Modified-Since:Tue, 24 Feb 2009 08:01:04 GMT,取值为上一次服务器返回的Last-Modified的值。
    服务端的头部是Last-Modified:Tue, 24 Feb 2009 08:01:04 GMT,表示资源修改的时间，如果服务器发现资源的最后一次修改时间晚于If-Modified-Since，就会返回新的资源（200 OK），否则，就不会返回资源的具体数据，而是只返回头部（304 Not Modified）
    缺点：在集群服务器上各个服务器上的文件时间可能不同；如果用旧文件覆盖新文件，因为时间更前，浏览器不会请求这个更旧的文件；时间精度为s级，对文件修改精度有严格要求的场景不能满足；客户端时间可能和服务端不同步。

  • Last-Modified与ETag是可以一起使用的，服务器会优先验证ETag，一致的情况下，才会继续比对Last-Modified，最后才决定是否返回304。

缓存流程

浏览器第一次请求得到的数据，通常会将一些静态资源缓存到本地，内存中或者磁盘中，并且响应头中会有这些缓存的过期时间。在过期时间之内，浏览器再次请求这些数据时会直接从本地缓存获取。如果超过过期时间，浏览器再次请求这些数据，服务器收到请求后，并不会立马将资源返回给客户端。而是先从请求头的一些信息中判断这次请求的资源有没有变化，如果没有变化，那么会返回给客户端一个：304 Not Modified状态码。告诉浏览器这次请求的资源没有变化，可以直接从缓存中获取，并且刷新缓存时间，如果变化了就返回200状态码和最新的资源。

无法被浏览器缓存的请求

  • 响应头禁止缓存，如Cache-Control:no-store，pragma:no-cache
  • fetch() 或 XMLHttpRequest 的特殊配置
  • 非 GET 请求
  • Set-Cookie 响应常常被浏览器特别对待，缓存行为受限
  • 某些响应类型或状态码，状态码非 200（如 500、403）通常不会被缓存；Content-Disposition: attachment 一般用于下载，默认不缓存；非 HTTP(s) 协议（如 FTP）或 blob、data URI 类型请求
  • 用户设置或模式限制缓存，浏览器开启隐身模式或用户或扩展禁用了缓存

不同用户行为的缓存行为

  • 强制刷新（ctrl+F5）：不使用强缓存，也不走协商缓存，全部重新下载资源，请求头中不包含If-Modified-Since和 If-None-Match，包含Pragma: no-cache和Cache-Control: no-cache
  • 普通刷新（F5）：先检查协商缓存，如果资源未变 → 返回 304，不重新下载
  • 链接点击/新开窗口/地址栏输入地址，检查强缓存，然后检查协商缓存，新开窗口在域名不一致的情况下应该不会使用内存缓存
  • 前进/后退：会命中Back-Forward Cache，用于快速恢复页面

从输入URL到页面展示

用户输入

  1. 如果是搜索内容，地址栏会使用浏览器默认的搜索引擎，来合成新的带搜索关键字的 URL。
  2. 如果判断输入内容符合 URL 规则，比如输入的是 time.geekbang.org，那么地址栏会根据规则，把这段内容加上协议，合成为完整的 URL。
  3. 当浏览器刚开始加载一个地址之后，标签页上的图标便进入了加载状态。但此时图中页面显示的依然是之前打开的页面内容，并没立即替换为极客时间的页面。因为需要等待提交文档阶段，页面内容才会被替换。

URL 请求过程

  1. 浏览器进程会通过进程间通信（IPC）把 URL 请求发送至网络进程，网络进程接收到 URL 请求后，会在这里发起真正的 URL 请求流程。
  2. 首先，网络进程会查找本地缓存是否缓存了该资源。
  3. 如果在缓存中没有查找到资源，那么直接进入网络请求流程。
  4. 请求前的第一步是要进行 DNS 解析，以获取请求域名的服务器 IP 地址。如果请求协议是 HTTPS，那么还需要建立 TLS 连接。
  5. 接下来就是利用 IP 地址和服务器建立 TCP 连接。连接建立之后，浏览器端会构建请求行、请求头等信息，并把和该域名相关的 Cookie 等数据附加到请求头中，然后向服务器发送构建的请求信息。
  6. 服务器接收到请求信息后，会根据请求信息生成响应数据，并发给网络进程。
  7. 在接收到服务器返回的响应头后，网络进程开始解析响应头，如果发现返回的状态码是 301 或者 302，那么说明服务器需要浏览器重定向到其他 URL，如果响应行是 200，那么表示浏览器可以继续处理该请求。
  8. 如果 Content-Type 字段的值被浏览器判断为下载类型，那么该请求会被提交给浏览器的下载管理器，同时该 URL 请求的导航流程就此结束。但如果是 HTML，那么浏览器则会继续进行导航流程。

准备渲染进程

默认情况下，Chrome 会为每个页面分配一个渲染进程，也就是说，每打开一个新页面就会配套创建一个新的渲染进程。但是，如果是同一站点，在某些情况下，浏览器会让多个页面直接运行在同一个渲染进程中

提交文档（CommitNavigation）

  • 接着网络进程接收到了响应头数据，便解析响应头数据，并将数据转发给浏览器进程。
  • 浏览器进程接收到网络进程的响应头数据之后，发送“提交导航 (CommitNavigation)”消息到渲染进程；消息包含响应信息和管道。
  • 渲染进程接收到“提交导航”的消息后，会通过管道和网络进程传输数据。
  • 渲染进程在接收到浏览器进程发来的导航提交请求（CommitNavigation）后，会开始加载新文档，并向浏览器进程发送“提交确认”消息（如 DidCommitProvisionalLoad）。该消息表明：导航已成功提交，文档已卸载并开始加载新内容，渲染进程已正式进入页面加载流程。
  • 浏览器进程在收到“确认提交”的消息后，会更新浏览器界面状态，包括了安全状态、地址栏的 URL、前进后退的历史状态，并更新 Web 页面

页面渲染

构建 DOM 树（Parse HTML）

  • 转码（Bytes -> Characters）
    读取接收到的 HTML 二进制数据，浏览器根据响应头或内容中的字符编码（如 UTF-8、GBK）将接收到的字节流解码为 HTML 字符串

  • Tokens 化（Characters -> Tokens）
    进行词法分析，词法分析器（分词器）会逐字符扫描输入，通过状态机来将 HTML 字符串转换为 词元（Token），每个 Token 代表 HTML 的语法单位（如开始标签、结束标签、文本节点等）

  • 构建 Nodes(Tokens -> Nodes)和 DOM 树（Nodes -> DOM Tree）
    树构造器根据解析得到的 Token 创建相应的 DOM 节点对象（如 Element、Text 等），设置节点属性，并按照文档结构建立节点之间的父子和兄弟关系。

构建 CSSOM 树

CSS的解析与 html 解析类似，最终将 CSS 文件解析为 StyleSheet 对象，每个对象包含一组 CSS 规则。每条规则由选择器和声明块组成，同时还包括与 CSS 语法结构对应的其他相关对象。

样式计算（Recaculate Style）
firefox与blink中的实现
  1. 规则匹配 & 分类（Selector Matching）
  • 解析 CSS 样式表后，将规则按选择器类型分类（ID、类、标签等），存入哈希映射表。
  • 遍历 DOM 元素时，仅从相关映射中提取可能匹配的规则。
  • 匹配的规则集合根据优先级、重要性、顺序等进行排序
  1. 构建规则路径和规则树（Rule Tree）
    排序好的规则可以构成一个规则路径，比如 body -> div。所有规则路径组成一个树结构。底部节点是优先级最高的规则（如行内样式、ID 选择器）。每个规则节点延迟计算，只有在需要该路径时才计算（lazy）。路径缓存下来，用于后续复用，提升性能。规则树中会保存自己规则的完整结构体（从最上层节点到自己节点为止的计算样式）
  2. 构建样式上下文树（Style Context Tree）
    每个 DOM 元素对应一个样式上下文（ComputedStyle）。
    样式上下文被分成多个结构体（如 Color, Border, Display, Font 等），每个结构体负责一类样式属性。结构体属性分为两种：继承属性（如 color）：若子节点没有定义，会从父节点继承。重置属性（如 display）：若未定义，则使用默认值。
    构建样式上下文时首先会计算规则树中的路径，或使用现有路径。然后开始应用路径中的规则，以填充新样式上下文中的结构体。
    如果末端路径节点中找到了部分定义，我们从路径的末端节点开始，向上遍历树，直到结构体填满为止。
    如果末端路径节点中某结构体未定义，则向上查找，直到找到该结构体并指向它的规则树节点（缓存共享）。
    如果最终找不到结构体的任何定义，那么如果结构体是“继承”类型，我们会在样式上下文树中指向父级的结构体。 如果它是重置结构体，则系统会使用默认值。
    如果计算后的节点有一些逻辑值，将逻辑样式值（如 %, em, inherit）转换为具体的物理值。
    然后，我们会将结果缓存在样式上下文树节点中，以便子节点使用。
    规则相同的兄弟元素之间可以共享样式上下文。

构建渲染树（Render Tree）

遍历 DOM 树中的所有可见节点，并把这些节点加到布局树中，而不可见的节点会被布局树忽略掉，比如 <script><meta> 或者display: none 的节点

布局（Layout/ Reflow）

遍历渲染树，确定页面各元素的位置、尺寸信息。

分层

因为页面中有很多复杂的效果，如一些复杂的 3D 变换、页面滚动，或者使用 z-indexing 做 z 轴排序等，为了更加方便地实现这些效果，渲染引擎还需要为特定的节点生成专用的图层，并生成一棵对应的图层树（LayerTree）收集这些分层
作用：提升性能，对于频繁动画或滚动的区域，如果整个页面都重绘性能开销会很大，分层后可以只重新合成动的部分（称为 GPU 合成）；支持异步操作，比如异步滚动（scrolling）、transform 动画，这些可以在主线程之外，由合成线程单独控制

图层绘制（Paint）

渲染引擎实现图层的绘制会把一个图层的绘制拆分成很多小的绘制指令，然后再把这些指令按照顺序组成一个待绘制列表

栅格化（raster）操作

绘制列表只是用来记录绘制顺序和绘制指令的列表，而实际上绘制操作是由渲染引擎中的合成线程来完成的。
在有些情况下，有的图层可以很大，比如有的页面你使用滚动条要滚动好久才能滚动到底部，但是通过视口，用户只能看到页面的很小一部分，所以在这种情况下，要绘制出所有图层内容的话，就会产生太大的开销，而且也没有必要。
基于这个原因，合成线程会将图层划分为图块（tile），这些图块的大小通常是 256x256 或者 512x512，如下图所示：

然后合成线程会按照视口附近的图块来优先生成位图，实际生成位图的操作是由栅格化来执行的。所谓栅格化，是指将图块转换为位图。而图块是栅格化执行的最小单位。渲染进程维护了一个栅格化的线程池，所有的图块栅格化都是在线程池内执行的，运行方式如下图所示：

通常，栅格化过程都会使用 GPU 来加速生成，使用 GPU 生成位图的过程叫快速栅格化，或者 GPU 栅格化，生成的位图被保存在 GPU 内存中。
相信你还记得，GPU 操作是运行在 GPU 进程中，如果栅格化操作使用了 GPU，那么最终生成位图的操作是在 GPU 中完成的，这就涉及到了跨进程操作。具体形式你可以参考下图：

从图中可以看出，渲染进程把生成图块的指令发送给 GPU，然后在 GPU 中执行生成图块的位图，并保存在 GPU 的内存中。

合成（composite）和显示

一旦所有图块都被光栅化，合成线程就会生成一个绘制图块的命令——“DrawQuad”，然后将该命令提交给浏览器进程。
浏览器进程根据 DrawQuad 命令，将其页面内容绘制到显存中，最后再将显存显示在屏幕上

页面渲染中外链资源的处理

  • CSS的下载和解析
    CSS 的下载和解析不会阻塞 DOM 树的构建。从 Chromium 61 开始，Blink 引入了一个名为 StyleThread 的专用线程，用于并行解析 CSS，从而提升性能。但由于渲染树（Render Tree）的构建依赖于 DOM 和 CSSOM，因此 CSS 的解析会阻塞渲染树的生成。此外，如果在 CSS 之后紧跟着有访问样式信息的 JavaScript，例如 getComputedStyle()，则 JavaScript 的执行也可能被阻塞，直到相关样式解析完成。

  • js的下载和解析
    JavaScript 的下载和解析通常会阻塞 DOM 树的构建，因为脚本可能会操作或依赖正在构建的 DOM，例如使用 document.write() 动态插入内容，或查询/修改元素。因此，当浏览器的 HTML 解析器遇到普通的 <script> 标签时，会暂停解析 HTML，直到脚本下载完成并执行完毕，然后再继续构建 DOM 树。
    非阻塞脚本有defer和async两种方式：defer脚本与 HTML 解析并行下载，在 DOM 树构建完成后、DOMContentLoaded 事件触发前统一执行，不会阻塞 DOM 解析，但能保证执行顺序与 HTML 中的顺序一致。async脚本与 HTML 解析并行下载，一旦下载完成就立即执行，可能会中断 DOM 构建，不保证执行顺序，但加载速度最快。
    向 document 动态添加 <script> 标签时，其 async 属性默认为 true。
    async 脚本可能在 DOMContentLoaded 事件之前或之后执行，但一定在 load 事件之前执行，因为 load 要等所有资源加载完成。

  • 图片的下载和解析
    当 HTML 解析器遇到 <img> 标签时，浏览器会执行两件事：

    1. 资源请求：立即解析 src 属性中的 URL。如果 URL 合法且图片尚未缓存，浏览器会将该资源交给网络栈进行异步下载（不会阻塞 HTML 解析）；若图片已被缓存，则可能直接跳过下载。

    2. 参与渲染流程：图片元素作为普通 DOM 节点继续参与构建渲染树、布局等流程。
      图片下载完成后，浏览器会将其从压缩格式（如 JPEG、PNG）解码为位图数据（bitmap）。此时如果图片没有预先设置尺寸，浏览器会根据解码得到的自然尺寸更新其布局信息。若新尺寸与原先布局不一致，将触发 reflow（回流），影响页面性能。

    因此，建议为 <img> 元素显式设置 width 和 height 属性，以便浏览器在首次布局时就能正确分配空间，避免因图片加载而引发不必要的 reflow。

  • 字体的解析

    1. 转码，根据响应头中的 Content-Type 或 声明的编码方式，将网页内容解码为 Unicode 字符流。
    2. 分段，浏览器将 Unicode 字符流按语言/书写系统（Script）进行分段，如：拉丁文、汉字、阿拉伯文、表情符号等。
    3. 字体匹配，根据css设置的font-family对每一段文字进行字体匹配，遵循字体的fallback机制
    4. 渲染，确定字体后，排版引擎根据文本、字体等参数生成字形和位置，然后调用不同平台下的rasterizer将字形转换成屏幕上的图案
页面渲染的优化措施
  • 预解析操作，当渲染引擎收到字节流之后，会开启一个预解析线程，用来分析 HTML 文件中包含的 JavaScript、CSS 等相关文件，解析到相关文件之后，预解析线程会提前下载这些文件。
  • 延迟样式阻塞
    从概念上讲，由于样式表不会更改 DOM 树，因此没有理由等待它们并停止文档解析。不过，在文档解析阶段，脚本会请求样式信息，如果样式尚未加载和解析，脚本将获得错误的答案，这显然会导致很多问题。如果有样式表仍在加载和解析中，Firefox 会挂起脚本等待。 而在WebKit 中只有当脚本尝试访问可能受未加载样式表影响的特定样式属性时，才会挂起脚本。
